总览与考试定位

M01 是理解后续所有模块的基础框架

⚡ 预计分值 ~8%
📌 本模块在考试中的定位：M01 不涉及复杂计算，考试题型以理解判断和排序分析为主。典型考法是给出某个渲染阶段，要求判断它在管线的什么位置、为什么在那里、如果在其他位置会有什么问题。

考试中 M01 相关题目约占 ~8%（约 1 道题）。核心考察点：

  • 管线阶段顺序：能按正确顺序写出 8 个阶段及各阶段所在的坐标系

  • 三大渲染范式区别：光栅化 (Geometry-first) vs 光线追踪 (Pixel-first) vs 实时渲染（光栅化基础 + 优化）
⚠️ 期中 Q8 已考过 Back-face Removal 顺序优势。期末可能以相似逻辑考查其他阶段的顺序理由，或与 GPU 可编程管线结合出题。

渲染管线的完整阶段

光栅化渲染管线的 8 个阶段，按执行顺序

⚡ 必考：默写管线顺序
📌 光栅化渲染管线是将 3D 场景逐阶段转换为屏幕上 2D 像素图像的有序处理流程。每个阶段对应一次坐标系或数据形式的变换。
1. Object Modeling 定义物体几何形状 · Local Space
2. World Transform 将各物体变换到统一世界坐标系 · Local → World
3. Perspective Transform 以 COP 为中心透视压缩，视锥体变长方体，保留 z · World → View
4. Back-face Removal 本页约定 nz > 0 → 背面，剔除约 50% 的面 · View Space
5. Clipping 裁剪视野外几何，6 平面已平行坐标轴 · View Space
6. Rasterization 图元 → 覆盖的像素集合 · Screen Space
7. Hidden Surface Removal Z-buffer 逐像素深度比较 · Screen Space
8. Shading Phong 光照模型计算每像素颜色 · Screen Space
⚠️ 考试容易混淆的点：透视变换（第 3 步，保留 z' = z）与透视投影（z' = d）不同。管线中用到的是透视变换（保留深度给后续 Back-face / Clipping / Z-buffer 使用），而不是透视投影（会丢失深度）。
按 Revision 原图默写：Input object models → World coordinate transformation → Perspective transformation → Back-face removal → Clipping → Rasterization → Shading → Output image。下方把 Rasterization 后的 image-space visibility（Z-buffer / Painter's）展开出来，是为了连接 M05 的隐藏面消除考点；若题目要求复述 Revision pipeline，按原图顺序写。

各阶段在 GPU 可编程管线中的执行位置（M09 关联）：

渲染阶段GPU 处理器可编程？
World Transform + Back-face RemovalVertex Shader（顶点着色器）✅
Clipping + 插值Rasterizer（光栅化器）❌ 固定功能
Z-buffer + Shading + 纹理Fragment Shader（片段着色器）✅

坐标系变换链

Object → World → View → Screen — 四跳坐标变换

⚡ 考试可能要求标注各阶段坐标系

从模型定义到屏幕像素，3D 数据经过四次坐标系变换：

变换从到操作管线阶段
Modeling TransformObject/Local SpaceWorld Space平移、旋转、缩放，建立物体间相对位置第 1 → 2 步
View TransformWorld SpaceView/Camera Space透视变换：视锥体 → 长方体，保留 z第 2 → 3 步
ProjectionView SpaceScreen/NDC Space将长方体映射到规范设备坐标（NDC）：x,y ∈ [-1,1]第 5 → 6 步
Viewport TransformNDC SpacePixel Coordinates映射到实际屏幕像素坐标Rasterization 内部
📌 记忆技巧：Object → World → View → Screen 四个空间，管线从第 1 到第 6 步依次完成这些变换。第 7–8 步都在 Screen Space 中执行。

Object Space

每个物体有自己的局部坐标系。球体中心在原点、立方体中心在原点，互不知道对方位置。

World Space

所有物体共享一个全局坐标系。球在 (0,0,0)，立方体在 (2,1,0)，它们的相对位置关系被建立。

View / Camera Space

以相机（COP）为中心的坐标系。透视变换在此空间将视锥体变为长方体，保留 z 深度。

管线顺序的设计逻辑

为什么 Back-face 先于 Clipping 先于 Rasterization？

⚡ 高频考点：Q8 变形

渲染管线的阶段顺序不是随意排列的，每个"前置"都有计算效率的考量——核心原则是越早淘汰不可见几何，后续阶段工作量越少。

顺序安排效率理由如果反过来会怎样
Back-face Removal 在 Clipping 之前 提前剔除约 50% 背面，减少 Clipping 需处理的面数 对大量最终不可见的背面浪费 Clipping 计算
Perspective Transformation 在 Clipping 之前 变换后 6 个裁剪平面变为平行于坐标轴的平面（x=±A, y=±B, z=d, z=C），裁剪退化为简单的坐标范围比较 需对任意方向的裁剪平面执行复杂的 3D 裁剪
Clipping 在 Rasterization 之前 不对不可见的几何体做光栅化 浪费光栅化资源处理视野外的几何体
Hidden Surface Removal 在 Shading 之前 只对可见像素计算光照，避免对被遮挡像素执行昂贵的 Phong 公式 对最终不可见的像素浪费着色计算
✅ Q8 标准答案框架（考场可直接套用）：

① 计算优势：对不透明物体约 50% 的面背向摄像机。在 Clipping 和 Rasterization 之前剔除这些面，避免了昂贵的裁剪计算和光栅化开销，大幅降低管线后续所有阶段的工作量。

② 若移到 Rasterization 之后：最终图像不会有差别（Z-buffer 最终也会丢弃被遮挡的片段），但极其低效——浪费了大量 GPU 资源在最终不可见的几何体上。

③ 设计哲学：从粗到细逐步筛选——Back-face Removal → Clipping → Rasterization → Z-buffer，越早剔除越高效。

三大渲染范式对比

光栅化 vs 光线追踪 vs 实时渲染

⚡ 理解型比较题
📌 这三种范式代表了不同的设计哲学：Geometry-first（光栅化，从三角形出发找像素）vs Pixel-first（光线追踪，从像素出发找几何）vs 在光栅化基础上的优化（实时渲染）。
维度Rasterization（光栅化）Ray Tracing（光线追踪）Real-time Rendering（实时渲染）
出发点 Geometry-first：逐三角形 → 像素 Pixel-first：逐像素 → 光线 → 场景 基于光栅化 + 优化技术
核心操作 三角形覆盖哪些像素？ 这个像素看到什么？ 如何在高帧率下保持可接受的画质？
速度 ✅ 极快（GPU 并行优化） ❌ 很慢（大量求交计算） ✅ 实时（通常 ≥30fps）
阴影 需额外算法（Shadow Map / Shadow Volume） ✅ Shadow Ray 自然处理 Shadow Map 为主
反射/折射 ❌ 难以精确处理 ✅ 递归追踪自然处理 近似（Screen Space Reflection 等）
全局光照 近似 / 需额外处理 镜面反射精确，间接漫反射需 Radiosity 预计算 + 近似
适用场景 游戏、实时交互、CAD 预览 电影特效、建筑可视化、离线渲染 3A 游戏、VR/AR、交互应用
⚠️ 范围提醒：Revision 明确排除的 Tutorial 扩展专题不需要复习；本页只保留三种会考渲染范式的比较。

Object Space vs Image Space 算法

考试可能要求区分这两种算法类型：

  • Object Space：在光栅化前处理 3D 几何体 → Back-face Culling
  • Image Space：在光栅化后处理像素 → Z-buffer、Painter's

考试常见陷阱

  • 把透视投影和透视变换搞混（前者丢 z，后者保留 z）
  • 认为 Back-face Removal 在 Clipping 之后
  • 认为管线阶段可以任意调换顺序
✍

考场答案模板

题型 1：解释 Back-face Removal 为何在 Clipping 之前执行（Q8 型）

答案框架：

① 基本事实：对不透明物体，约 50% 的面背向摄像机。Back-face Removal 可在 O(n) 时间内剔除它们。

② 计算优势：提前剔除 50% 的面 → Clipping 阶段处理的几何量减半 → 显著降低后续所有阶段的计算量。

③ 如果反过来：若 Clipping 先执行，需对所有面（含 50% 最终不可见的背面）执行裁剪计算，严重浪费。

④ 最终图像：即使 Back-face Removal 移到 Rasterization 之后，最终图像也相同（Z-buffer 会丢弃背面），但效率极低。

题型 2：默写渲染管线阶段及坐标系

答案框架（Revision 原图 + 展开说明）：

① Input Object Models (Local Space) → ② World Coordinate Transformation (Local → World) → ③ Perspective Transformation (World → View) → ④ Back-face Removal (View) → ⑤ Clipping (View, 轴对齐) → ⑥ Rasterization (Screen) → ⑦ Shading → ⑧ Output Image。

补充说明：Hidden/Back Surface Removal 是 Revision 的独立考点；其中 Z-buffer / Painter's 属于 rasterization 后的 image-space visibility 处理，答管线原图时不要把它写乱顺序。

题型 3：比较三种渲染范式

答案框架（用出发点一句话区分）：

① Rasterization：Geometry-first，逐三角形 → 像素，速度快但阴影/反射需额外算法。

② Ray Tracing：Pixel-first，逐像素 → 光线 → 场景求交，阴影/反射/折射自然处理但极慢。

③ Real-time Rendering：在光栅化基础上加入 LOD、Shadow Map 等优化，达到实时帧率。

自测题

Q1. 透视变换（Perspective Transformation）和透视投影（Perspective Projection）的关键区别是什么？

答：透视变换保留 z 坐标不变（z' = z），将视锥体变为长方体，为后续 Back-face Removal 和 Clipping 保留深度信息。透视投影丢弃深度信息，z' = d（投影平面位置），用于最终成像。

Q2. 为什么透视变换要放在裁剪之前执行？

答：透视变换前，视锥体是斜截锥体（frustum），6 个裁剪平面是任意方向的平面，裁剪计算复杂。透视变换后，视锥体变为轴对齐长方体，6 个裁剪平面变为 x=±A、y=±B、z=d、z=C（平行于坐标轴），裁剪退化为简单的坐标范围检查。

Q3. Back-face Removal 和 Z-buffering 分别属于 Object Space 还是 Image Space 算法？

答：Back-face Removal = Object Space（处理 3D 几何体，在光栅化前），检查法向量 z 分量判断正/背面。Z-buffering = Image Space（处理像素/片段，在光栅化后），逐像素比较深度值。Painter's Algorithm 也是 Image Space。

Q4. 如果把 Back-face Removal 从管线中完全移除，最终渲染图像会改变吗？为什么？

答：对不透明物体，最终图像不会有差别——因为 Z-buffer 在后续阶段会通过逐像素深度比较丢弃被遮挡的背面片段。但性能会严重下降：需要处理所有面（含背面）的变换、裁剪、光栅化、着色。对透明物体，可能会看到背面透过，图像可能不同。

Q5. 光栅化渲染管线中，哪个阶段将数据从 3D 连续表示转换为 2D 离散像素表示？

答：Rasterization（光栅化）。它将图元（三角形、线段等）转换为屏幕像素网格中的离散片段（fragments）。这是管线中从"几何表示"到"图像表示"的关键转折点。在 GPU 管线中，光栅化器（Rasterizer）是固定功能硬件，还负责对顶点属性做插值。

必背：Revision 原图管线顺序
