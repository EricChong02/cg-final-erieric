LOD 技术（Level of Detail）

根据物体与观察者的距离动态调整几何细节，近处高精度、远处低精度

⚡ Discrete vs Progressive 对比
📌 核心概念：LOD 的核心思想是用距离换精度——远距离物体的细节人眼无法分辨，用低精度模型渲染可以大幅节省 GPU 计算资源，而视觉质量几乎不受影响。

离散 LOD（Discrete LOD）

预处理阶段：为每个物体预计算若干个不同分辨率的模型（如 LOD0=高, LOD1=中, LOD2=低）

运行时：根据物体到视点的距离，直接选择最合适的 LOD 级别切换

优点缺点
运行时高效，直接选择预计算模型即可需要决定「需要几个 LOD 级别」——不好确定
实现简单切换 LOD 时出现视觉跳变（Popping Artifact）
需存储多个完整模型，存储开销大

渐进网格（Progressive Mesh）

Progressive Mesh 是连续 LOD 方法——通过 Edge Collapse（边折叠）逐步降低分辨率，或通过 Vertex Split（顶点分裂）逐步恢复。不是预存多个离散级别，而是存储操作序列。

对比维度Discrete LODProgressive Mesh
LOD 切换跳变（Popping）平滑连续
存储多个完整模型base mesh + 操作序列
运行时开销低（直接选模型）中（需执行 vsplit 操作）
适用场景场景物体数量少，切换频率低大型连续场景（地形等）
⚠️ 考试重点：Discrete LOD 的最大问题是 Popping（视觉跳变），Progressive Mesh 用连续 vsplit 操作解决了这个问题。答题时必须点出「连续」vs「离散」的对比。

Edge Collapse & Vertex Split

Progressive Mesh 的核心操作——一面减少、一面恢复

⚡ 高频考点：减少/增加 2 面

Edge Collapse（边折叠）

将一条边的两个端点合并为一个顶点

  • 与该边相邻的两个三角形被删除
  • 面数 减少 2，顶点数减少 1
  • 符号：Mn →ecol Mn-1

Vertex Split（顶点分裂）

将一个顶点分裂为两个相邻顶点，引入新边

  • 新增 2 个三角形
  • 面数增加 2，是 Edge Collapse 的逆操作
  • 符号：Mn-1 →vsplit Mn

Progressive Mesh 结构与存储格式

高分辨率原始模型 Mori = Mn
  ↓ ecol × n 次（逐步降低）
低分辨率基础网格 M0（base mesh）

存储格式：base_mesh (M0) + vsplit_sequence [vs0, vs1, ..., vsn-1]
每个 vsplit 记录：分裂哪个顶点、新顶点位置、新三角形的连接关系
📌 两大应用优势：
① 高效存储：只需存 base mesh + vsplit 序列，比存储所有离散 LOD 层次大幅节省空间
② 渐进式传输：网络传输时先发 base mesh（低分辨率）立即可用，再逐步发送 vsplit 序列精化细节（渐进式加载，用户体验好）
⚠️ 闭卷必背数字：Edge Collapse = 减少 2 个面；Vertex Split = 增加 2 个面。考试会直接问"从 M0 恢复高分辨率的操作是什么？"——答：连续执行 Vertex Split。

Selective Refinement（选择性细化）

只对视点关注的局部区域进行高分辨率细化，其他区域保持低分辨率

⚡ Progressive Mesh 的扩展
📌 核心思想：在大型模型中，不需要全局统一分辨率。近处/注视点附近用高精度，远处/视野边缘用低精度。这是 Progressive Mesh 的自然扩展——不同区域可以独立决定是否执行 vsplit。

核心挑战：相邻区域的依赖关系

Mesh 相邻区域之间有强依赖关系，不能随意在某区域做 Vertex Split 而不影响邻居，否则会产生：

  • T-junction（T 形连接）：高分辨率区域与低分辨率区域交界处的顶点不匹配
  • Crack（裂缝）：相邻区域分辨率不一致导致网格出现缝隙

解决方法：维护"半规则网格"或"适应性细化"结构，在边界处做特殊处理以消除裂缝和 T 形连接。根据视距、屏幕投影大小动态调整各区域的细化级别。

⚠️ 考试角度：Selective Refinement 是 Progressive Mesh 的扩展应用，考试中通常作为 Progressive Mesh 题的加分点或延伸问法。理解"局部细化 vs 全局细化"的概念即可，不要求写出具体算法。

视觉质量与渲染代价的估计

如何判断一个物体的 LOD 级别是否合适

⚡ 影响因子判断

视觉质量影响因素

因素影响逻辑
物体的 LOD 级别（三角形数量）三角形越多 → 视觉质量越高
物体到视点的距离越远 → 屏幕投影越小 → 可以用更低精度
物体在屏幕上的投影尺寸投影越小 → 所需精度越低
物体的运动速度快速运动 → 人眼无法看清细节 → 可用低精度
物体与视线方向的角距离偏离注视点 → 人眼不敏感 → 可降精度

渲染代价估计

渲染代价主要由以下因素决定（从高到低影响）：

  • 三角形数量（主要因素，每帧 GPU 需处理的图元数）
  • 所用着色方法：Flat < Gouraud < Phong（每像素计算开销递增）
  • 纹理复杂度（纹理采样、过滤开销）
  • 阴影计算（Shadow Map 需额外渲染一遍，Shadow Volume 需生成阴影体几何）
⚠️ 考试角度：可能以判断题或选择题出现——"以下哪种情况下物体可以用较低 LOD？"正确选项通常指向：物体距离远 / 运动速度快 / 不在注视中心。

阴影技术

Hard Shadow vs Soft Shadow；Shadow Volume（Stencil Buffer）vs Shadow Map（两遍渲染）

⚡ 必考对比题

Hard Shadow vs Soft Shadow

Hard Shadow（硬阴影）

  • 阴影边界锐利，无过渡
  • 由点光源产生
  • 只有本影（Umbra），无半影
  • 实现方法：Shadow Volume / Shadow Map

Soft Shadow（软阴影）

  • 阴影边界渐变，从暗到亮过渡
  • 由面光源（Area Light）产生
  • 包含本影（Umbra）+ 半影（Penumbra）
  • 实现更复杂（多采样等）

Shadow Volume（阴影体）

Step 1以光源为视点，找到物体的轮廓边（Silhouette Edges）
Step 2从轮廓边向光源方向延伸，构建阴影锥体（Shadow Volume）
Step 3Stencil Buffer 计数法：从视点向像素画线，穿过阴影体正面 +1，穿过背面 −1
Step 4最终计数 > 0 → 像素在阴影中；计数 = 0 → 被光照亮
优点缺点
阴影在屏幕空间完全精确（像素级，无走样）需要额外处理阴影体几何，方法复杂
处理开放场景（光线离开场景）需要特殊处理
软阴影扩展困难

Shadow Map（阴影贴图）

Pass 1以光源为视点渲染整个场景，生成深度图（Shadow Map）
Pass 2从观察者视角正常渲染，对每个像素 P 变换到光源空间
判断zP > zmap → P 被遮挡（阴影中）；zP ≤ zmap → P 可见（被照亮）

Shadow Volume vs Shadow Map —— 闭卷必背对比表

维度Shadow VolumeShadow Map
精度像素精确（无走样）受 Shadow Map 分辨率限制（有走样）
额外几何需要生成阴影体几何无需额外几何
实现复杂度较复杂（需处理边缘情况）较简单（额外一次渲染）
软阴影扩展困难容易（多采样组合 Shadow Map）
动态场景中（需更新阴影体）高（每帧重新渲染 Shadow Map）
适用场景静态场景，需要精确阴影动态场景，游戏中广泛使用
主要限制额外 Polygon 处理复杂走样（Aliasing）
📌 Shadow Map 走样的改进方法 —— Perspective Shadow Map（PSM）：先对场景从观察者角度做透视变换，使近处物体在 Shadow Map 中占据更多像素（更高分辨率），再生成 Shadow Map。这样近处物体的阴影更精确，远处物体的精度降低（但人眼不敏感）。
⚠️ 考试陷阱：Shadow Map 的主要限制是走样（Aliasing），不是精度低。Shadow Volume 是像素精确的，但代价是需要额外构建几何体。这两种方法的 trade-off 是考试的经典考点。

GPU vs CPU

数据级并行（同一操作不同数据）vs 任务级并行（流水线）

⚡ 两种并行必考

CPU vs GPU 核心差异

对比维度CPUGPU
核心数量少量强大核心（4–64 个）大量简单核心（数千个）
优化目标低延迟（Latency-oriented）高吞吐量（Throughput-oriented）
适合任务控制密集型（Control-heavy）数据密集型（Data-heavy）
数据级并行弱强（SIMD，同一操作作用于不同数据）
任务级并行中（少量线程）强（Pipeline 流水线，多阶段并发）

GPU 的两种并行类型（闭卷必背）

数据级并行（Data-level Parallelism）

核心 对不同数据执行相同操作

例 每个三角形可以独立处理，互不依赖

实现 GPU 将 10,000 个三角形分配给 10,000 个核心同时并行处理顶点变换

任务级并行（Task-level Parallelism）

核心 对同一数据流的不同阶段并发执行

例 渲染管线分多个阶段，每阶段独立运行

实现 顶点着色器处理第 n 帧时，光栅化器处理第 n−1 帧，片段着色器处理第 n−2 帧

📌 一句话记忆：数据级并行 = "同样的活，分给不同的人同时干"（三角形独立）；任务级并行 = "流水线，各工序同时干不同批次的活"（管线阶段重叠）。

Efficient Communication

GPU 不只要算得快，还要避免 CPU-GPU 之间频繁搬数据

⚡ Revision 明确列出
📌 核心思想：GPU 的高吞吐优势只有在数据能连续留在 GPU 端处理时才明显。若每一步都要在 CPU 和 GPU 之间来回传输，通信开销会抵消并行计算收益。
原则考场解释对应管线例子
批量传输 一次传大量顶点/纹理数据，比频繁小块传输更高效 把 mesh buffer、texture buffer 预先上传到 GPU
数据驻留 中间结果尽量留在 GPU 内存，不反复读回 CPU 顶点处理、光栅化、片段处理在 GPU 流水线内连续完成
减少同步 CPU 等 GPU 或 GPU 等 CPU 都会破坏流水线并行 避免每帧等待单个 draw call 完成后再提交下一步
闭卷一句话：Efficient communication means minimizing data transfer and synchronization between CPU and GPU, keeping data on the GPU so data-level parallelism and pipeline/task-level parallelism can actually pay off.

可编程 GPU 管线

顶点着色器（几何变换、背面消除）→ 光栅化器（裁剪、插值）→ 片段着色器（Z-test、着色、纹理）

⚡ 高频：三处理器职责
📌 从固定功能到可编程：传统硬件图形管线（Fixed Function Pipeline）的所有阶段行为固定，开发者无法修改。可编程管线（Programmable Pipeline）引入着色器（Shader），开发者可自定义顶点处理和片段处理两个关键阶段的行为。

三阶段职责划分（闭卷必背表）

处理器可编程？主要任务
顶点着色器
(Vertex Shader)		 可编程 ① 几何变换（模型/视图/投影矩阵变换）
② 背面消除（Back-face Removal）
③ 逐顶点光照计算（Gouraud Shading 在此）
光栅化器
(Rasterizer)		 固定功能 ① 裁剪（Clipping）
② 顶点表示 → 像素/片段（光栅化）
③ 对逐顶点属性（颜色、纹理坐标、法向量）在像素间插值
片段着色器
(Fragment Shader)		 可编程 ① 深度测试（Z-test / Depth Comparison）
② 逐像素颜色计算（Phong Shading 在此）
③ 纹理采样（Texture Sampling / Lookup）
⚠️ 高频陷阱：注意区分两个着色器中的光照——Gouraud Shading 在顶点着色器（逐顶点计算光照），Phong Shading 在片段着色器（逐像素计算光照）。光栅化器是固定功能（不可编程），不要写错。
📌 核心对应关系速记：
· 变换 / 背面消除 → 顶点着色器
· 插值 → 光栅化器
· Z-test / 着色 / 纹理 → 片段着色器

计算机动画三种方法

Keyframe（关键帧）/ Procedural（过程动画）/ Motion Capture（动作捕捉）

⚡ 必考对比表

Keyframe Animation（关键帧动画）

动画师指定关键时刻（Keyframes）的物体状态（位置/旋转/缩放/颜色/光照），计算机通过插值（线性或样条如 Bezier、Catmull-Rom）自动生成中间帧（In-betweens）。

Procedural Animation（过程动画）

由算法/物理规则/模拟自动生成运动。典型应用：粒子系统（烟雾、火焰）、刚体动力学（碰撞响应）、群体行为（鸟群 Boids、人群模拟）。

Motion Capture（动作捕捉）

用传感器/摄像头记录真实运动者的3D 运动数据，映射到虚拟角色上。典型应用：电影特效（如 Gollum）、体育游戏（FIFA）、格斗游戏。

三种动画方式完整对比（闭卷必背表）

维度关键帧动画 (Keyframe)过程动画 (Procedural)动作捕捉 (Motion Capture)
原理 动画师设定关键时刻状态，计算机插值中间帧 用算法/规则/物理仿真自动生成运动 记录真实演员/物体的 3D 运动数据并应用到角色
控制精度 高（艺术家精确控制） 低（由规则和初始条件决定） 中（记录真实动作，后期调整有限）
真实感 取决于动画师技能 高（物理仿真自然） 极高（真人动作数据）
制作效率 低（手工设定关键帧） 高（算法自动生成） 高（实时录制）
设备要求 普通动画软件 物理引擎/粒子系统 专业动捕设备（昂贵）
适用场景 角色动画、表情动画、可控情节 布料模拟、流体、烟雾、群体 写实人物动作（格斗、运动员、表情捕捉）
主导方式 艺术家主导 算法主导（规则/物理） 真实数据驱动
⚠️ 考试角度：三种方式的核心区别——Keyframe → 艺术家主导；Procedural → 算法主导（规则/物理）；MoCap → 真实数据驱动。动捕技术细节考试不深究，了解概念和光学动捕（Optical MoCap）即可。

考场答案模板

题型 1：GPU 可编程管线三处理器职责（P9-1）

答案框架：

① 顶点着色器（可编程）：负责几何变换（模型/视图/投影矩阵）、背面消除（Back-face Removal）、逐顶点光照计算（Gouraud Shading）

② 光栅化器（固定功能）：负责裁剪（Clipping）、顶点→片段转换、逐顶点属性插值（颜色/纹理坐标/法向量）

③ 片段着色器（可编程）：负责深度测试（Z-test）、逐像素颜色计算（Phong Shading）、纹理采样

④ 数据级并行：对不同数据执行相同操作——10,000 个三角形同时分配给 10,000 个核心并行处理

⑤ 任务级并行：流水线模式——顶点着色器处理第 n 帧时，光栅化器处理第 n−1 帧，片段着色器处理第 n−2 帧

⑥ Efficient communication：减少 CPU-GPU 数据传输和同步，让数据留在 GPU 管线内连续处理。

题型 2：Efficient Communication 为什么重要

答案框架：

① GPU 擅长高吞吐并行计算，但 CPU-GPU 通信相对昂贵。

② 若频繁把顶点、纹理或中间结果在 CPU 和 GPU 之间来回传输，会产生额外延迟，并破坏 GPU 流水线并行。

③ 高效策略是批量上传数据、让数据驻留在 GPU memory、尽量在 GPU 管线内部完成连续阶段，并减少同步等待。

④ 结论：efficient computation 依赖 data-level / task-level parallelism；efficient communication 保证这些并行收益不会被数据搬运抵消。

题型 3：Shadow Volume vs Shadow Map 对比（P9-2）

答案框架：

① Shadow Volume：从光源找到轮廓边 → 延伸构建阴影锥体 → Stencil Buffer 计数（正面+1，背面−1，最终>0 即阴影中）。精度：像素精确，无走样。限制：需要生成额外阴影体几何，复杂。

② Shadow Map：Pass 1 从光源渲染深度图 → Pass 2 从观察者比较深度（zP > zmap = 阴影）。精度：受 Shadow Map 分辨率限制。限制：走样（Aliasing），改进为 PSM。

③ 游戏常用 Shadow Map（简单、适合动态场景）；精确需求用 Shadow Volume。

题型 4：三种计算机动画方式对比（P9-3）

答案框架：

① Keyframe（艺术家主导）：动画师设定关键状态，计算机插值。适合角色动画、表情动画。缺点：手工耗时。

② Procedural（算法主导）：物理规则/模拟驱动，自动生成运动。适合布料、流体、群体。缺点：难以精确控制。

③ Motion Capture（数据驱动）：记录真人 3D 运动数据。适合写实人物动作。缺点：设备昂贵。

题型 5：Progressive Mesh 操作与存储（P9-4）

答案框架：

① Edge Collapse：合并一条边的两个顶点 → 删除相邻 2 个三角形 → 面数减少 2

② Vertex Split：分裂一个顶点为两个 → 新增 2 个三角形 → 面数增加 2（ecol 的逆操作）

补充题. Efficient communication 在 GPU 渲染中为什么重要？

参考答案：GPU 的优势来自大量核心并行处理和图形管线流水线，但 CPU-GPU 之间传输数据和同步等待都很昂贵。如果每个阶段都把中间结果传回 CPU，再传回 GPU，通信开销会抵消 GPU 并行计算的收益。

高效做法：批量上传顶点/纹理等数据；尽量让数据驻留在 GPU memory；把顶点处理、光栅化、片段处理放在 GPU 管线内连续完成；减少 CPU 等 GPU 或 GPU 等 CPU 的同步点。

一句话：Efficient computation 让 GPU 算得快；efficient communication 保证 GPU 不把时间浪费在搬数据和等待上。

P9-4. Progressive Mesh 的操作与存储

解释 Edge Collapse 和 Vertex Split 操作的含义，说明 Progressive Mesh 如何利用这两个操作实现高效存储和渐进式网格传输。

参考答案

Edge Collapse（边折叠）：将一条边的两个端点合并为一个顶点 → 相邻 2 个三角形被删除 → 面数减少 2，顶点数减少 1。表示为 Mn →ecol Mn-1。

Vertex Split（顶点分裂）：将一个顶点分裂为两个相邻顶点，引入新边 → 新增 2 个三角形 → 面数增加 2。表示为 Mn-1 →vsplit Mn。是 Edge Collapse 的逆操作。

Progressive Mesh 结构：Mn → Mn-1 → ... → M1 → M0（base mesh）。

存储格式：base_mesh (M0) + vsplit_sequence [vs0, vs1, ..., vsn-1]

两大应用优势：

  • 高效存储：base mesh + vsplit 序列比所有 LOD 层次省空间
  • 渐进式传输：先发低模立即可用，再逐步发送精化数据，用户体验好
📌 核心优势：连续 LOD（无跳变） + 高效存储（base mesh + 操作序列）
